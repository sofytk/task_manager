## Постановка задачи

Целью проекта является разработка консольного приложения для управления задачами с поддержкой:

* создания задач
* редактирования и обновления статуса задач
* сортировки задач по приоритету и категории
* фильтрации задач по дате
* сохранения задач

Каждая задача должна содержать:

* дату выполнения
* название
* описание
* категорию
* приоритет
* статус выполнения

Система должна обеспечивать удобное взаимодействие через консольный интерфейс и хранить данные в файле (JSON).

---

## Анализ задачи и структура классов

### Анализ задачи

Для реализации системы управления задачами необходимо:

1. **Хранение данных**
   Используется JSON-файл как простая файловая база данных.

2. **Разделение логики**
   Программа разделена на несколько уровней:

   * работа с данными (Data)
   * бизнес-логика (Domain)
   * пользовательский интерфейс (Presentation)

3. **Моделирование сущностей**
   
   Основная сущность — задача (`Task`), которая содержит все необходимые поля.

4. **Использование перечислений**
   
   Для категорий и приоритетов применяются `Enum`, что исключает ошибки ввода.

---

### Структура классов

#### 1. `Task`

Модель задачи.

**Поля:**
* id - уникальный индификатор задачи
* date — дата
* title — название
* description — описание
* priority — приоритет (Enum)
* category — категория (Enum)
* is_done — статус выполнения

**Методы:**

* `to_dict()` — преобразование в словарь
* `from_dict()` — создание объекта из словаря

---

#### 2. `Category` (Enum)

Содержит категории задач:

* Работа
* Дом
* Семья
* Здоровье и др.

---

#### 3. `Priority` (Enum)

Содержит уровни приоритета:

* высокий
* средний
* низкий

---

#### 4. `BaseTaskRepository` (абстрактный класс)

Определяет интерфейс для работы с данными:

* `get_all_tasks()`
* `get_tasks_by_date(date)`
* `add_task(task)`
* `update_data(task)`
* `delete_task(task)`

---

#### 5. `TaskRepository`

Реализация работы с JSON-файлом.

**Функции:** Наследует от `BaseTaskRepository`

---

#### 6. `TaskController`

Слой бизнес-логики, связывает `TaskRepository` c `Client`

**Функции:**
* `add_task(task)` - добавление новой задачи
* `delete_task(task)` - удаление выбранной задачи
* `update_task_status(task)` - обновление статуса задачи
* `get_all_tasks()` - получение всех задач
* `get_today_tasks()` - получение задач на сегодня
* `get_task_by_date(date)` - получение задач на выбранную дату
* `sorted_by_priority()` - сортировка по приоритету
* `sorted_by_category()` - сортировка по категориям

---

#### 7. `Client`

Консольный интерфейс пользователя.

Функции и методы:
* `show_menu()` – основной цикл меню для выбора действий пользователя.
* `_add_task()` – создание и добавление новой задачи с вводом данных пользователем.
* `_show_all_tasks()` – вывод всех задач из репозитория.
* `_show_today_tasks()` – вывод задач на текущую дату.
* `_show_tasks_by_date()` – фильтрация задач по указанной дате.
* `_sort_by_priority()` – сортировка задач по приоритету с группировкой.
* `_sort_by_category()` – сортировка задач по категориям с группировкой.
* `_mark_task_done()` – пометка выбранной задачи как выполненной.
* `_delete_task()` – удаление выбранной задачи.
* `format_task(task, index=None)` – форматирование задачи в строку для вывода на экран.
* `choose_enum(enum_type)` – выбор значения из перечисления (Priority или Category).
* `input_date(prompt="...")` – запрос и валидация даты у пользователя.
* `display_tasks(tasks)` – вывод списка задач на экран.

## Тестирование

Также добавлены классы `TestTaskRepository` и `TestTaskRepositoryIntegration` для тестирования `TaskRepository` - компонента, отвечающего за хранение и управление задачами в JSON-файле. Тестирование включает в себя **unit-тесты** (с использованием мок-объектов) и **интеграционные тесты** (с реальной файловой системой).

---

### 1. Unit-тесты (`TestTaskRepository`)

Unit-тесты изолируют тестируемый компонент от внешних зависимостей (файловой системы) с помощью мок-объектов. Это позволяет проверить логику работы репозитория без реального доступа к диску.

#### Тестовые сценарии:

| № | Название теста | Описание |
|---|----------------|-----------|
| 1 | `test_get_all_tasks_success` | Проверка успешного получения всех задач из файла |
| 2 | `test_get_all_tasks_empty_file` | Проверка обработки пустого файла |
| 3 | `test_get_all_tasks_with_invalid_data` | Проверка пропуска некорректных записей |
| 4 | `test_get_tasks_by_date` | Проверка фильтрации задач по дате |
| 5 | `test_get_tasks_by_date_no_matches` | Проверка фильтрации при отсутствии совпадений |
| 6 | `test_add_task_success` | Проверка добавления новой задачи |
| 7 | `test_delete_task_success` | Проверка удаления существующей задачи |
| 8 | `test_delete_task_not_found` | Проверка удаления несуществующей задачи |
| 9 | `test_update_task_success` | Проверка обновления существующей задачи |
| 10 | `test_update_task_not_found` | Проверка обновления несуществующей задачи (исключение) |

---

### 2. Интеграционные тесты (`TestTaskRepositoryIntegration`)

Интеграционные тесты проверяют взаимодействие репозитория с реальной файловой системой. Для каждого теста создается временная директория с изолированным JSON-файлом, которая автоматически удаляется после завершения теста.

#### Тестовые сценарии:

| № | Название теста | Описание |
|---|----------------|-----------|
| 1 | `test_integration_get_all_tasks` | Проверка получения всех задач из реального файла |
| 2 | `test_integration_add_and_get` | Проверка добавления задачи и последующего получения |
| 3 | `test_integration_update_task` | Проверка обновления существующей задачи |
| 4 | `test_integration_delete_task` | Проверка удаления задачи |
| 5 | `test_integration_get_by_date` | Проверка фильтрации задач по дате |

---

python3 -m unittest test.task_repository_test
